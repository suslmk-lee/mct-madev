================================================================================
MCT-MADEV: Multi-Agent Orchestration System with 3D Office UI
초기 프롬프트 (Initial Reproduction Prompt)
================================================================================

이 문서는 MCT-MADEV 프로젝트를 다른 환경에서 동일하게 재현하기 위한
완전한 프롬프트입니다. AI 어시스턴트에게 이 문서를 제공하면 전체 시스템을
처음부터 구축할 수 있습니다.

================================================================================
1. 프로젝트 개요
================================================================================

MCT-MADEV는 LLM 기반 멀티 에이전트가 가상 3D 오피스에서 협업하는
오케스트레이션 시스템입니다.

핵심 기능:
- CEO(사용자)가 채팅으로 지시 → PM 에이전트가 작업 분해 → 팀원 에이전트가 실행
- 3D 오피스 UI에서 에이전트가 걸어다니고, 상태 변화를 시각적으로 표현
- 실시간 WebSocket 기반 상태 동기화
- 7개 LLM 프로바이더 지원 (Anthropic, OpenAI, Google, Ollama, Kimi, MiniMax, GLM)
- CLI로 서버 시작, 에이전트/스킬 관리

기술 스택:
- TypeScript (strict, ESM only)
- PNPM monorepo (packages/* + apps/*)
- Express + WebSocket (서버)
- React 19 + Three.js (React Three Fiber) + Zustand (프론트)
- sql.js (인메모리 SQLite)
- Biome (린트/포맷)
- tsup (번들링)
- Vite + Tailwind CSS v4 (웹 빌드)

3.2 오케스트레이터 (src/orchestrator/)

5. packages/db — SQLite 데이터베이스
================================================================================

■ IDatabase 인터페이스 (types.ts):
  Projects, Agents, Tasks, Workflows, Messages, Logs, TokenUsage CRUD

■ SqliteDatabase (sqlite/SqliteDatabase.ts):
  - sql.js 사용 (인메모리 + 디스크 영속화)
  - PRAGMA foreign_keys = ON
  - 7개 테이블 스키마:

  CREATE TABLE projects (
    id TEXT PRIMARY KEY, name TEXT NOT NULL, description TEXT,
    repo_path TEXT, config TEXT DEFAULT '{}',
    created_at TEXT, updated_at TEXT
  );

  CREATE TABLE agents (
    id TEXT PRIMARY KEY, project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    name TEXT NOT NULL, role TEXT NOT NULL, provider TEXT NOT NULL, model TEXT NOT NULL,
    system_prompt TEXT, visual_state TEXT DEFAULT 'IDLE',
    position TEXT DEFAULT '{"x":0,"y":0,"z":0}',
    current_task_id TEXT, metadata TEXT DEFAULT '{}',
    created_at TEXT, updated_at TEXT
  );

  CREATE TABLE tasks (
    id TEXT PRIMARY KEY, project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    workflow_id TEXT REFERENCES workflows(id),
    parent_task_id TEXT REFERENCES tasks(id),
    title TEXT NOT NULL, description TEXT DEFAULT '', status TEXT DEFAULT 'CREATED',
    assignee_agent_id TEXT REFERENCES agents(id),
    priority INTEGER DEFAULT 0, dependencies TEXT DEFAULT '[]',
    metadata TEXT DEFAULT '{}', result TEXT, error TEXT,
    created_at TEXT, updated_at TEXT
  );

  CREATE TABLE workflows (
    id TEXT PRIMARY KEY, project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    name TEXT NOT NULL, definition TEXT NOT NULL,
    status TEXT DEFAULT 'PENDING', current_stage_id TEXT,
    results TEXT DEFAULT '{}', created_at TEXT, updated_at TEXT
  );

  CREATE TABLE messages (
    id TEXT PRIMARY KEY, task_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    agent_id TEXT NOT NULL REFERENCES agents(id),
    role TEXT NOT NULL, content TEXT NOT NULL, token_count INTEGER,
    created_at TEXT
  );

  CREATE TABLE token_usage (
    id TEXT PRIMARY KEY, project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    agent_id TEXT NOT NULL REFERENCES agents(id),
    provider TEXT NOT NULL, model TEXT NOT NULL,
    input_tokens INTEGER DEFAULT 0, output_tokens INTEGER DEFAULT 0,
    created_at TEXT
  );

  CREATE TABLE logs (
    id TEXT PRIMARY KEY, project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    level TEXT NOT NULL, source TEXT, message TEXT NOT NULL,
    metadata TEXT DEFAULT '{}', created_at TEXT
  );

  ** 중요: deleteAgent() 시 FK 참조 먼저 정리:
     DELETE FROM token_usage WHERE agent_id = ?
     DELETE FROM messages WHERE agent_id = ?
     UPDATE tasks SET assignee_agent_id = NULL WHERE assignee_agent_id = ?
     DELETE FROM agents WHERE id = ?

■ chat.ts — CEO↔PM 채팅 (핵심 플로우):

  POST /projects/:projectId/chat

  ① 3단계 의도 분류 (Intent Classification):

    Tier 1 — 키워드 사전 판별 (keywordClassify):
      DIRECTIVE_PATTERNS (45개 정규식):
        한국어: 개발|구현|만들어|생성|제작|작성|빌드|수정|변경|삭제|추가|배포|
                테스트|해줘|해주세요|하세요|합시다|하자|페이지|컴포넌트|기능|
                API|서버|DB|데이터베이스
        영어: implement|develop|build|fix|deploy|test|feature|api
      CHAT_PATTERNS (4개): 인사, 감사, 질문, 일상
      판정: 2+ directive 키워드 → 0.85 confidence (directive)
            1 directive (no chat) → 0.7 confidence
            chat 패턴만 → 0.75 confidence (chat)

    Tier 2 — LLM 분류 (confidence < 0.7인 경우):
      시스템 프롬프트로 한국어/영어 의도 분류 요청
      JSON 응답 파싱: {"intent":"chat"|"directive","confidence":0.0-1.0}

    Tier 3 — 최종 폴백:
      키워드 결과 사용 또는 기본값 "chat"
      긴 메시지(>20자) + uncertain → directive 쪽으로

  ② directive 판정 시:
    - 루트 태스크 생성 (parentTask)
    - PM.decompose() → 서브태스크 분해
    - assignSubtasks() → 라운드로빈 배분
    - executeSubtasks() → 비동기 실행 + 채팅 진행 상황 broadcast

  ③ chat 판정 시:
    - PM이 대화로 응답 (최근 20개 메시지 컨텍스트)

  ④ assignSubtasks() — 라운드로빈 부하 분산:
    - 역할별 에이전트 그룹화
    - 배치 내 할당 횟수 추적, 최소 할당 에이전트 우선
    - 같은 역할 에이전트 간 균등 배분

  ⑤ detectBestRole() — 키워드 기반 역할 감지:
    DEVELOPER: implement, code, develop, frontend, backend, 개발, 구현, 코드...
    REVIEWER: review, check, 리뷰, 검토...
    TESTER: test, qa, 테스트, 검증...
    DEVOPS: deploy, ci, docker, 배포, 인프라...
    PM: plan, manage, 기획, 관리...

  ⑥ ensureAgentForRole() — 필요한 역할 에이전트 자동 생성

  ⑦ executeSubtasks() — 비동기 실행 플로우:
    각 서브태스크마다:
      1. 에이전트 찾기 (없으면 자동 생성)
      2. agent.visualState → WORKING
      3. task.status → IN_PROGRESS
      4. SkillLoader.selectSkills() → 관련 도구 선택
      5. LLM 호출 (도구 포함)
      6. Tool-use 루프 (최대 5 반복): 도구 호출 → 결과 → 다음 호출
      7. task.status → DONE/FAILED
      8. agent.visualState → IDLE
      9. 채팅 진행 상황 broadcast

  ⑧ broadcastChatStatus() — 채팅 상태 메시지:
    시작: "[1/N] Task Title 시작 → Agent Name"
    성공: "✓ Task Title 완료 (Agent Name)\nPreview..."
    실패: "✗ Task Title 실패 (Agent Name)\nError..."
    요약: "전체 작업 완료: X/N 성공, Y/N 실패"

14. 핵심 아키텍처 패턴
================================================================================

1. 이벤트 드리븐: EventEmitter → WebSocket broadcast → Zustand 반영
2. DB 추상화: OrchestratorDeps/IDatabase로 구현체 분리
3. 프로바이더 패턴: IProvider로 7개 LLM 통합
4. 상태 머신: TaskStateMachine으로 유효한 태스크 전이만 허용
5. DAG 실행: TaskDAG로 의존성 순서 보장
